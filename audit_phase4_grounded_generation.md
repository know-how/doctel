# Phase 4: Grounded Generation Design Audit — DocTel LLM Pipeline

**Date**: 2026-07-16
**Scope**: Full citation enforcement chain from system prompt → RAG context → LLM call → response delivery
**Status**: 3 findings (P0, P1, P1) — NO post-generation verification exists anywhere in pipeline

---

## Executive Summary

The DocTel LLM pipeline has **strong citation instructions in the system prompt** and **correct RAG context embedding in most code paths**, but **zero defense-in-depth**. There is no post-generation verification layer — no groundedness check, no citation validation, no hallucination detection. The system relies entirely on the LLM's willingness to follow instructions, which is fundamentally unreliable.

The single worst offender is the **unsafe fallback** (Issue #2, P0) which sends the system prompt's citation-demanding rules with **zero RAG context**, forcing the LLM to fabricate sources to satisfy its own instructions. Additionally, Issue #14 (P1) — a wasted LLM call in the streaming endpoint — doubles API costs and increases hallucination surface.

### Issue #15 (P0-CRITICAL): No Post-Generation Groundedness Verification — Entire Pipeline

**Root Cause**: The DocTel LLM pipeline has no defense-in-depth for grounded generation. After the LLM generates a response, no component verifies that:
1. The answer actually uses the provided context
2. Citations in the answer correspond to real chunks in the context
3. The answer does not contain hallucinated facts
4. The answer does not contradict the provided context

The system relies entirely on prompt engineering ("Use ONLY the provided context") — but LLMs are known to ignore or deviate from such instructions, especially when the system prompt is very long (~150 lines for ZETDC persona) and citation rules are buried within formatting rules, summary standards, and organizational context.

**Exact Evidence**:
- `ask.py` lines 62-175 (`_stream_cloud_answer`): Error classification exists (timeout, connection, rate_limit, etc.) but **zero output validation** — every successful response is passed through as-is
- `ask.py` lines 640-700 (ask_global response building): `rag["answer_text"]` from non-grounded fallback is saved directly to DB without any factual consistency check
- `rag_service.py` lines 292-330 (system prompt with citation rules): Rules are instructions only — no enforcement mechanism
- `provider_gateway_service.py`: Records `InteractionAudit` with input/output but has no output quality validation
- `app/services/opencode_zen_service.py` lines 420-560: Returns raw LLM output without any verification

**Exact Fix** — Add a groundedness verification layer. There are several approaches:

**Option A (Lightweight — Rule-based)**:
Add a post-generation check in `_stream_cloud_answer()` and the non-streaming paths that verifies:
- If context was provided, the answer must reference at least one source filename
- If context was provided but answer contains claims not supported by context, flag as ungrounded

Implementation in `app/services/groundedness_check.py`:
```python
# New file: app/services/groundedness_check.py

import re
import logging
from typing import List, Optional

logger = logging.getLogger(__name__)

def verify_citation_grounding(
    answer_text: str,
    context_filenames: List[str],
) -> dict:
    """
    Verify that citations in the answer correspond to actual context sources.
    Returns {"passed": bool, "issues": List[str], "citation_count": int, "grounded_citations": int}
    """
    if not context_filenames:
        # No context provided — can't verify grounding
        return {"passed": True, "issues": [], "citation_count": 0, "grounded_citations": 0}
    
    issues = []
    # Extract all [Source: ...] citations from answer
    source_pattern = r'\[Source:\s*([^,\]]+)(?:,\s*Chunk\s*(\d+))?\]'
    matches = re.findall(source_pattern, answer_text)
    
    for filename, _ in matches:
        filename = filename.strip()
        # Check if this filename appears in the provided context
        if not any(fname and filename.lower() in fname.lower() for fname in context_filenames if fname):
            issues.append(f"Citation refers to unknown source: '{filename}'")
    
    return {
        "passed": len(issues) == 0,
        "issues": issues,
        "citation_count": len(matches),
        "grounded_citations": len(matches) - len(issues),
    } if matches else {
        # No citations found — could be acceptable if no context
        "passed": True,
        "issues": [],
        "citation_count": 0,
        "grounded_citations": 0,
    }
```

**Option B (LLM-based — LLM-as-Judge)**:
After generation, send the question + context + answer to a small, cheap LLM (e.g., `qwen3:4b` locally) with a prompt like:
```
Given the question, context, and answer below, determine if the answer is
fully grounded in the provided context. Answer only: GROUNDED or NOT_GROUNDED.
If NOT_GROUNDED, list one fact that is not supported.

Question: {question}
Context: {context}
Answer: {answer}
```
This could be integrated into `_stream_cloud_answer()` at `ask.py:62-175` or into the response builder at `ask.py:640-700`.

**Before Behaviour**:
- LLM receives strict citation instructions in system prompt
- LLM generates response with citations that may be fabricated
- Response is saved and displayed to user WITHOUT any verification
- Users see quotations from non-existent documents as if they were real

**After Behaviour**:
- After generation, a groundedness check runs automatically
- Flagged responses are either rejected (retry with stronger grounding prompt) or prefixed with a confidence warning
- Citations that reference non-existent sources are detected and stripped
- Logs record grounding quality metrics for monitoring

### Issue #16 (P1-HIGH): Unsafe Fallback Creates Systematic Hallucination Machine (Issue #2 Reaffirmed)

**Root Cause**: The unsafe fallback in `ask_global()` (lines 537-625) sends `settings.zetdc.system_prompt` — which contains strict citation rules ("Use ONLY the provided context", "Never invent or paraphrase quotes", "ALWAYS include the exact quoted text") — but provides **zero RAG context**. This forces the LLM into an impossible position: either disobey its own instructions, or fabricate citations from thin air.

**Exact File**: `app/routers/ask.py`
**Exact Lines**: 556-560 (gateway_generate call with raw question + citation-demanding system prompt)

```python
answer_text, reasoning_text_fb = await asyncio.wait_for(
    gateway_generate(db, question, model_id=chosen_model, system=sys_prompt),
    timeout=120.0,
)
```

Where `sys_prompt = (settings.zetdc.system_prompt or "").strip() or None` (line 542) and `question` is the raw user question (line 555).

Also at lines 571-572:
```python
answer_text = await ollama.generate(chosen_model, question, system=sys_prompt)
```

And the model_router fallback at lines 583-589:
```python
routed = await asyncio.wait_for(
    select_model_with_fallback(question, task_type="rag"),
    timeout=120.0,
)
```

**Exact Fix**: Replace the unsafe fallback with an honest no-context response. The `rag_service.py` NO_KNOWLEDGE_FOUND guardrail (lines 264-290) already implements this correctly:

```python
return {
    "answer_text": "I could not find any relevant information in the knowledge base to answer your question. Please try rephrasing or providing more specific details.",
    ...
    "used_model": "no-knowledge-found",
}
```

The fix is to make `ask_global()` respect this guardrail return rather than overriding it:

In `ask.py` lines 537-625, replace the entire unsafe fallback block with:
```python
if not rag or not rag.get("citations"):
    # ── NO_KNOWLEDGE_FOUND guardrail — do NOT call LLM without context ──
    logger.warning(
        "[RAG_GUARD] No RAG context available — returning NO_KNOWLEDGE_FOUND "
        "instead of calling LLM (question=%r, scope=%s, project_ids=%s)",
        question[:120], scope, project_ids,
    )
    rag = {
        "answer_text": "I could not find any relevant information in the knowledge base to answer your question. Please try rephrasing or providing more specific details.",
        "reasoning_text": "",
        "citations": [],
        "cross_references": [],
        "used_model": "no-knowledge-found",
    }
    # ── END GUARDRAIL ──────────────────────────────────────────────────
```

**Before Behaviour**: When RAG returns no citations, the system calls the LLM with the raw question and a citation-demanding system prompt. The LLM fabricates plausible-sounding but entirely fake document citations. Users are misled into believing documents contain information they do not.

**After Behaviour**: When RAG returns no citations, the system returns an honest "I could not find any relevant information" message. No LLM call is made. No hallucinated citations are generated. The user is invited to rephrase or provide more detail.

### Issue #17 (P1-HIGH): Wasted LLM Call in Streaming Endpoint Doubles Hallucination Surface (Issue #14 Reaffirmed)

**Root Cause**: `ask_global_stream()` calls `get_rag_answer_scoped()` purely to extract citations/chunks, but `get_rag_answer_scoped()` internally calls the LLM to generate `answer_text`. This `answer_text` is discarded — the streaming endpoint then calls the LLM AGAIN via `_stream_cloud_answer()`. The first LLM call is completely wasted, doubling API costs and giving the LLM two opportunities to hallucinate.

**Exact File**: `app/routers/ask.py`
**Exact Lines**: 793-820 (RAG retrieval call for streaming)

```python

    project_ids,
    question,
    db,
    document_id=None,
    model_name=chosen_model,

# ... extracts citations_list from rag_result ...
# rag_result["answer_text"] is NEVER USED
```

Then at line ~888:
```python
async for event in _stream_cloud_answer(augmented_question, chosen_model, sys_prompt, ...):
```

**Exact Fix**: Create a retrieval-only function that does NOT call the LLM. The existing `rag_service.py` function `get_rag_answer_scoped()` needs to be split or refactored to expose a "retrieve only" path.

**Option A (Minimal change)** — Add a `retrieval_only=True` parameter to `get_rag_answer_scoped()`:

In `app/services/rag_service.py`, add parameter to the function signature:
```python

    model_name=None, force_policy=False, force_diagram=False,
    retrieval_only=False,  # <-- NEW

```

And before the LLM call (around line 310-330), add:
```python

    return {
        "answer_text": "",  # No LLM call
        "reasoning_text": "",
        "mermaid_code": "",
        "drawing_prompt": "",
        "citations": citations,
        "cross_references": cross_refs,
        "used_model": "retrieval-only",
        "embedding_mismatch": embedding_mismatch,

```

Then in `ask.py` line 793, change to:
```python

    project_ids, question, db, document_id=None,

```

**Option B (Cleaner)** — Create a dedicated retrieval function:
```python
# In app/services/rag_service.py
async def retrieve_context(
    project_ids, user_query, db, document_id=None
) -> dict:
    """Retrieve relevant chunks via ChromaDB — NO LLM call.
    Returns {citations: [...], context: str, cross_references: [...]}
    """
    # ... same embedding + ChromaDB logic as get_rag_answer_scoped ...
    # ... build context + citations ...
    # ... NO LLM call, NO guardrail check that needs LLM ...
    return {"citations": citations, "context": context, "cross_references": cross_refs}
```

**Before Behaviour**: Streaming endpoint calls LLM once to generate `answer_text` that is discarded, then calls LLM again for the actual stream. 2 LLM calls per streaming question, double API cost, double hallucination surface.

**After Behaviour**: Streaming endpoint uses a retrieval-only function that extracts context from ChromaDB without any LLM call. The single LLM call via `_stream_cloud_answer()` is the only generation. API cost halved, hallucination surface reduced.

## System Prompt Citation Rules: Full Text

For reference, the citation rules embedded in the ZETDC system prompt (`config.yaml` lines 74-220+):

```
1. When citing information, ALWAYS include the exact quoted text from the source in quotation marks.
2. Format citations like: [Source: filename, Chunk N] immediately after the quote.
3. Example: According to the manual, 'Application for reconnection must be submitted...' [Source: Dunning Manual.pdf, Chunk 13].
4. Never invent or paraphrase quotes - only use exact text from the provided sources.
5. Multiple citations should each have their own quoted text.
```

The RAG service prompt also has:

```
Use ONLY the provided context to answer.
```

And the streaming endpoint builds:

```
Answer the question using ONLY the provided context. Always cite sources.
```

**These rules are strong instructions but have zero enforcement.** An LLM that decides to be "helpful" by generating an answer (even without context) will violate these rules, and nobody will know.

## Recommended Architecture: Grounded Generation Pipeline

```
User Question
    │
    ├──→ Intent Detection (greeting? → skip RAG)
    │
    ├──→ RAG Retrieval
    │       ├──→ Embedding → ChromaDB/pgvector search
    │       └──→ NO_KNOWLEDGE_FOUND? → RETURN honest message (NO LLM CALL)
    │
    ├──→ Build Augmented Prompt
    │       ├──→ System prompt + citation rules
    │       ├──→ Context from retrieval
    │       └──→ User question
    │
    ├──→ LLM Generation
    │       └──→ gateway.generate() / gateway.generate_stream()
    │
    └──→ POST-GENERATION VERIFICATION (NEW — Issue #15)
            ├──→ Citation Grounding Check
            │       └──→ Extract [Source: ...] from answer
            │       └──→ Verify each source exists in retrieved context
            ├──→ Factual Consistency Check (LLM-as-Judge)
            │       └──→ Is answer supported by context?
            └──→ Output Decision
                    ├──→ PASS → Return to user
                    ├──→ FAIL (minor) → Log warning + return with confidence caveat
                    └──→ FAIL (major) → Retry with stronger grounding prompt or return NO_KNOWLEDGE_FOUND
```
